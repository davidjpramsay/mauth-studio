# AI Chatbox Readiness

This is the practical path from the current Codex-driven workflow to an in-app ChatGPT-like assistant that can inspect and edit Mauth documents directly.

## Goal

The product goal is that the in-app assistant becomes at least as capable for Mauth document work as the current Codex-in-repo workflow. A teacher should eventually be able to ask inside Mauth for the same kind of help currently handled in this thread: create and edit tests, write and repair solutions, adjust student space, build diagrams, fix formatting, inspect print/layout problems, manage files, and update the document safely.

The in-app assistant should not manipulate DOM nodes, local component state, or hidden editor details. It should call explicit Mauth tools, receive structured results, and let the editor commit accepted changes through the same undo, autosave, and file-version system used by human edits.

Parity does not mean giving the model unrestricted browser or filesystem control. It means exposing enough intentional Mauth tools, inspections, validators, repair loops, and document context that the assistant can reach the same result through safer app-native actions.

## Parity Target

The in-app assistant should be treated as the primary AI workflow, not a convenience side panel. It should be able to:

- Understand the current document, selected item, file path, solution visibility state, print mode, page count, and relevant renderer warnings.
- Create complete assessment content: title-page details, questions, parts, subparts, marks, answer spaces, diagrams/charts, worked solutions, mark ticks, and validation passes.
- Make iterative edits from natural language, including "make Q4 harder", "fix this spacing", "add a diagram", "write the solutions", "check every question has enough space", and "combine these two tests".
- Inspect and repair output quality, including missing solutions, insufficient answer space, diagram label issues, page overflows, print/PDF problems, and inconsistent formatting.
- Work with files like a normal project assistant: list, open, save, duplicate, rename, move, delete, restore versions, and eventually search across projects.
- Accept teacher-provided context such as screenshots, pasted images, PDFs, Word documents, curriculum notes, marking requirements, and example questions.
- Use rule brains and deterministic validators as its finishing standard, not vibes or raw text confidence.
- Explain final changes in normal teacher language while hiding raw tool JSON, internal ids, and provider payloads.

## Implemented Baseline

- The Mauth action layer supports deterministic document edits for questions, parts, subparts, modules, solution slots, marks, diagrams, page breaks, front matter, logos, page format, formatting config, and validation.
- Dry-run previews return a proposed document and structured summary before a change is committed.

  - `mauth.author.addDiagram`
  - `mauth.author.ensureSolutions`
- The assistant file-tool layer in `apps/web/src/lib/mauthAssistantFileTools.ts` exposes project-file operations:

  - `mauth.files.open`

- The assistant adapter in `apps/web/src/lib/mauthAssistantAdapter.ts` dispatches document and file tool calls, commits accepted document edits through a host callback, parses opened files through a host callback, serialises current documents for save/save-as, and reports changed ids/paths through plain chat replies.
- `mauth.question.upsert` is the high-level one-question authoring tool. For focused requests such as "write question 1 with a solution" or "make question 2" when the document currently ends at Question 1, the provider should use this compact payload instead of manually emitting many low-level module edits. The tool replaces an existing question or appends the immediate next missing question. It builds question text, optional diagram blocks, optional structured parts, student-only answer space, solution-only solution text, marks, and item order, validates the generated content, applies the update, and lets the adapter commit it through history/autosave.
- For screenshot/PDF/Word one-question conversions, `mauth.question.upsert` should preserve the visible source structure in one payload: stem text, native question-level diagram blocks when the source has a mathematical diagram, then structured parts with non-empty part text. A prompt like "add this question to the test" without an explicit number means source-question authoring for the immediate next missing question, not a diagram-only follow-up on an existing question. Do not convert a visible source diagram into prose such as "The diagram shows...", and do not create blank part prompts after moving their expressions into the stem.
- When an attachment-backed one-question conversion explicitly asks for the diagram/graph/chart to be entered, placed underneath, or kept from the source, the backend exposes a stricter `mauth_question_upsert` schema that requires `diagram`. This prevents the provider from accepting a text-only replacement as a successful conversion.
- Screenshot conversions should not invent worked solutions just because the source lists marks. Unless the prompt asks for answers/solutions or the source visibly includes them, omit `solutionText`, `parts[i].solutionText`, and `includeSolution`.
- For scalar-product/vector-ray screenshots with magnitudes and angle markers but no coordinate axes, the assistant should prefer the compact `vectorRayDiagram` diagram field. It accepts source-friendly vectors with `id`, `length`, `angleDeg`, optional label positions, magnitude labels, and angle markers, then expands deterministically into native `vector2d` with axes/grid hidden. If the assistant hand-writes raw `vector2d`, store each ray as a common-origin vector in `metadata.vector2d.vectors` using source-faithful rectangular components. Every vector entry must include `id`, `name`, `start:[x,y]`, and `components:[dx,dy]`; `segmentLabels.vectorId` and `angleMarkers.from`/`to` must reference those vector ids. Angle-marker endpoints must match the source semantics: if the right-angle square marks `b` perpendicular to `d`, use `from:"b", to:"d"`; if the `45^\circ` arc is between `c` and `d`, use `from:"c", to:"d"`. Use `labelStyle: "custom"` with labels such as `\mathbf{a}`, preserve magnitude labels with `metadata.vector2d.segmentLabels`, and preserve right-angle/numeric angle markings with `metadata.vector2d.angleMarkers`. Use Penrose `geometricConstruction` for theorem geometry, not source ray diagrams where the source placement must be preserved.
- In high-level question authoring, structured part prompt text is rendered as the first visible text module inside that part, followed by the student answer space and any solution module. This is deliberate because the preview renders part content blocks; the stored `part.text` field is metadata/summary only.
- Successful high-level authoring calls can finish locally when no semantic review is required: after solution-only or response-space commits, the chat panel can show the local result message instead of making a second provider call just to say the edit is done. Diagram-bearing edits are different: the adapter returns a compact `semanticReview` packet with the focused question preview and actual diagram summary, and the provider must compare the teacher request, question text, and diagram payload before reporting success. If it sees a mismatch, such as a question about two straight lines but a quadratic graph payload, it should repair with the focused high-level tool rather than asking the teacher to notice.
- The visible Assistant panel is available only in display-only mode from a small left-side floating toggle. It has a normal chat layout with messages above and the input fixed at the bottom, executes model tool calls through `runMauthAssistantAdapterTool`, commits accepted edits through editor history/autosave, updates active file state for file tools, and hides tool plumbing from the teacher. Do not expose raw tool JSON, internal tool names, ids, or provider payloads in the teacher-facing panel.
- Assistant chat state is scoped to the active project file path. Switching files should load that file's own assistant conversation instead of carrying stale chat history from another test. Unsaved documents share a temporary unsaved conversation until they are saved to a project path.
- Fresh assistant intent routing and provider input should be based on the latest teacher message, current attachments, and tool outputs, not stale earlier chat turns. This prevents an old "add a diagram" exchange from hijacking a later "add this attached question to the test" request, and it keeps token spend lower for screenshot/PDF conversions. Repair continuations may still include the relevant tool output so the model can fix the exact failed payload.
- Long assistant runs should show calm teacher-facing activity labels such as Thinking, Inspecting document, Previewing changes, Applying changes, Checking document, Opening file, or Saving file, with a small elapsed timer. These labels are status only; they must not expose tool JSON or require the teacher to manage internal edit steps.
- Active-file saves are revision-guarded. The adapter passes the active file path and loaded revision to file tools, and the frontend stores that revision in recovery autosave. Assistant/manual saves must not fetch the latest revision immediately before saving, because that would bypass external-change protection.
- Save and revision friction should be handled by the assistant/API in the background wherever safe. The teacher should not see raw revision numbers, file-tool chatter, or repeated conflict messages. If the current file is dirty before a file open/move/duplicate/rename, save it through the loaded revision first. If a Codex/script/API edit changes the active project file, update the active autosave draft and revision state as part of the same workflow. Ask the teacher only when choosing automatically could lose work, and then offer a concise reload-or-save-as-copy style choice.
- If a provider response reaches the frontend tool-round limit while it still has pending function calls, store that pending `responseId` and tool-call list. A later “continue” message must resume by sending the missing tool outputs for those calls, not by starting a normal user-message request with the pending `previous_response_id`; OpenAI will reject that as a missing function-call output.
- The backend provider boundary is `GET /api/assistant/status` and `POST /api/assistant/chat`. The OpenAI adapter lives in `apps/api/app/services/openai_assistant.py`, uses the Responses API, sends compact rule-brain context from `configs/ai-brains/`, exposes direct high-level authoring tools for common teacher prompts, and keeps the generic `mauth_tool` wrapper for broader document/file tools. Tool calls are mapped back to real Mauth tool names and passed through to the frontend adapter.
- Fresh teacher prompts now use a small brain-menu planner before the authoring call. The planner receives the teacher prompt, a compact document summary, and the available brain menu, then calls `mauth_select_brains` with only the needed packs such as `question`, `diagram`, `solutions`, and `formatting`. The authoring call receives those selected brains rather than the whole rule set. If the planner fails, the backend falls back to deterministic keyword selection so the assistant still works.
- The backend exposes direct provider tools for focused teacher tasks, then maps them back to real Mauth tools before the frontend adapter sees them. `mauth_question_upsert` and `mauth_convert_source_question` map to `mauth.question.upsert`; `mauth_make_diagram_for_question` maps to `mauth.author.addDiagram`; `mauth_write_solutions_for_questions` maps to `mauth.author.ensureSolutions`; `mauth_fix_question_formatting` maps to `mauth.format.apply`. `mauth_author_replace_question`, `mauth_author_add_diagram`, and `mauth_author_ensure_solutions` remain as legacy aliases for older calls.
- For focused single-question prompts, the backend exposes only the narrow direct provider tool where possible. For example, "add the diagram to Question 1" exposes `mauth_make_diagram_for_question` without the broad wrapper tool, and "make Question 1 from this screenshot" exposes `mauth_convert_source_question`. A prompt like "add this question to the test" is treated as source-question authoring and targets the immediate next missing question when no question number is stated. This keeps the model on the intended path and reduces token spend.
- The backend now classifies fresh teacher requests before provider routing. If an edit request is missing the target or source, such as "add a diagram" with no question number, or "add this question to the test" with no attached/pasted source, it returns one short clarifying question without an OpenAI tool round. Clear requests still route to the narrow native Mauth tool.
- `mauth.author.addDiagram` is the placement/replacement tool for focused diagram follow-ups. The assistant should choose a renderer and provide a real `graphConfig`, not a canned diagram recipe. Use Penrose `geometricConstruction` for schematic geometry and circle theorem diagrams; use `graph2d` for coordinate/function graphs, `vector2d` for coordinate vectors, `statsChart` for statistical charts, `setDiagram` for Venn diagrams, `graph3d` for 3D diagrams, and `image` for uploads. For source `graph3d` solids such as coordinate prisms, include named `data.points`, visible `data.segments`, and renderer-native `metadata.view3d` values with `az`, `el`, and `bank` in radians; camera-only placeholders, degree-like camera values, segment `style`, nested `camera.eye` metadata, and axis helper points such as `xAxis`/`yAxis`/`zAxis` are not enough. `standardDiagram` recipe names are not supported for assistant-authored diagrams. Repair calls can pass `diagramId` to replace an existing bad diagram.
- For source slope-field/direction-field questions, `graph2d` can store `data.slopeField` with an expression such as `(x - 1) / (2*y)`, grid values or ranges/steps, and `highlightedPoints` for any explicitly requested point slope. Solution curves should be normal `graph2d.functions` entries; when the source solution is an implicit equation, prefer a `kind: "relation"` entry instead of separate `sqrt(...)` branches. The assistant should not fake slope fields as prose or unrelated line-segment features.
- `graph2d` validation now rejects common plausible-but-ignored nesting mistakes before apply: `graphConfig.data.functions`, `graphConfig.data.features`, direct `graphConfig.data.xRange/yRange`, axes/size fields inside `graphConfig.options`, function `domain`, function `style` objects, feature `type`, feature `style` objects, and invented region fields such as `expressionTop`, `expressionBottom`, `opacity`, and `fillColor`. Graph bounds, size, display flags, functions, and features must be top-level graphConfig fields; function intervals use `domainMin`/`domainMax`, functions style with `color`/`strokeWidth`/`strokeStyle`, and features use `kind` plus direct `color`/`size`/`strokeWidth` fields. Shaded graph2d regions should reference boundary functions through `region_between_curves`, `region_curve_axis`, or `region_clipped_by_curve` indices and use `fillOpacity`.
- High-level assistant diagram fields now have a strict native block contract: diagram items must be shaped as `{ graphConfig: { type: ... }, diagramAlign?: ... }`, or `{ vectorRayDiagram: { vectors: [...] }, diagramAlign?: ... }` for source scalar-product ray diagrams. Top-level renderer fields such as `diagram.type`, `diagram.data`, or `diagram.options`, and alias fields such as `diagram.config`, are rejected before apply with repairable `arguments.diagram.graphConfig` validation issues. Incomplete `vectorRayDiagram` payloads are also rejected before apply so the model repairs vector ids, directions, labels, and angle-marker references instead of creating a fallback diagram.
- High-level assistant diagram authoring also runs deterministic intent validation before applying edits. It catches obvious wrong-renderer choices such as circle/tangent/chord geometry sent as `graph2d`, scalar-product ray diagrams sent as `network`/`graph2d`/`geometricConstruction`, coordinate component-vector diagrams sent as Penrose geometry, probability charts sent as `graph2d`, Venn diagrams sent as the wrong renderer, and network prompts sent as coordinate/vector geometry. These failures return repairable `arguments.diagram.graphConfig.type` issues and no document mutation.
- The statistics renderer intent now includes probability density functions, normal curves, and sample-mean distribution sketches, not only histograms and probability bars. These should use `statsChart`; `graph2d` remains for ordinary coordinate/function graphs. Within `statsChart`, use `chartType: "density"` for arbitrary source density curves, `chartType: "normal"` for parameterised normal curves, and `chartType: "blankAxes"` for axes students sketch on.
- For Penrose geometry, the normal assistant path is now native supported Penrose Substance in `graphConfig.options.substanceSource`. The model should choose `geometricConstruction`, receive the compact Diagram Brain/Penrose cheat sheet, then write declarations such as `Point`, `Circle`, `Line`, `Ray`, `NamedSegment` and predicates such as `CircleThrough`, `OnCircle`, `Tangent`, `Segment`, `VectorSegment`, `ParallelToSegment`, `PerpendicularToSegment`, and `RightAngle`. `Segment` and `VectorSegment` are predicates, not types: declare `NamedSegment AB, beam` first, then call `Segment(AB, A, B)` or `VectorSegment(beam, L, P)`. Never write standalone declarations such as `Segment AB` or `VectorSegment beam`. Labels use declaration syntax too: write `Label A $A$`, not `Label(A, $A$)`. Structured `graphConfig.data` geometry remains available for simple UI-driven cases, but it is not the primary AI geometry language.
- Before assistant-authored Penrose diagrams are committed, the Mauth tool boundary checks common unsupported Substance patterns such as `LabelsPoint`, `SegmentLength`, `OppositeRays`, `Collinear`, `Connect`, raw TeX inside `LabelsAngle(...)`, non-parenthesized predicate calls, `LabelSegment`, raw `Ray(...)`, and bad `RightAngle`/`PerpendicularToSegment` arity. The frontend adapter then preflight-renders changed `geometricConstruction`, `setDiagram`, and `network` blocks through the same `/api/diagram/penrose` request used by preview. After render succeeds, assistant commit preflight also blocks changed Penrose circle-theorem diagrams whose declared geometry does not match the prompt. These failures return repairable `validationIssues` and no document mutation occurs; the assistant should repair the `graphConfig` rather than leaving a wrong or failed diagram in the document.
- `mauth.preview.inspect` now reports a diagram-level `warnings` array on each `question.diagrams[]` entry. These warnings combine renderer intent checks, missing image sources, missing scalar-product vector labels/angle markers, statsChart probability/relative-frequency mistakes, vector2d missing named coordinate vectors, graph3d missing source vertices/segments, setDiagram missing requested shading/counts, graph2d missing requested tangent/region/asymptote/slope-field features, browser-rendered diagram failures when preview metrics are mounted, likely label collisions, clipping/page-fit warnings, and conservative Penrose circle-geometry semantic warnings. Use them after diagram creation/repair before telling the teacher a diagram is correct.
- After successful high-level diagram edits, the assistant receives a compact semantic review artifact rather than a raw full document dump. It includes the focused question/module text previews, part summaries, diagram renderer, expressions/features/data/substance summary, structural warnings, and rendered metrics when available. This is where model intelligence should be used: compare the requested task with the actual artifact and repair mismatches. Do not keep adding narrow deterministic rules for every possible graph/question pairing when a compact semantic review can catch the issue more generally.
- `mauth.preview.inspect` still reports conservative `semanticWarnings` for Penrose circle-theorem diagrams. Use them after diagram creation/repair to catch valid-but-wrong geometry such as missing `Tangent`, missing `ParallelToSegment` to the named chord, missing chord segments, named points not on the intended circle, or visible auxiliary labels that the question does not name.
- The adapter now runs an automatic post-edit inspection after high-level diagram commits. It waits for the browser preview to repaint when mounted, then runs `mauth.preview.inspect` with fresh rendered metrics. If inspection finds repairable diagram or rendered-layout warnings, the tool result is deliberately returned as failed with `validationIssues` and the target `diagramId` where available; the provider should retry the same high-level diagram tool once with that `diagramId` or repair the adjacent layout, then only report success after the repaired diagram passes inspection.
- Before any accepted assistant document edit is committed, the frontend adapter also runs document preflight for common high-cost mistakes. It blocks accidental deletion of existing shared diagrams outside genuine question replacement, blocks changed diagrams with hard diagram-inspection failures such as wrong renderer, missing image source, missing scalar-product vector labels/angle markers, invalid manual probability data, missing named vector2d metadata, missing requested set shading/counts, or invalid Penrose circle geometry, and blocks changed solution containers that keep visible mark notes, omit the standard `**Solution.**` heading, or have hidden `[[marks:n]]` totals that do not match the relevant marks. Failures return repairable `validationIssues` and no document mutation occurs.
- The frontend tool loop gives failed tool/preflight output one provider repair attempt using the exact validation issues. If the repair also fails, the loop stops with a clean teacher-facing message instead of continuing through expensive failed internal edit steps.
- Repair continuations stay on the same narrow direct authoring tool where possible. A failed `mauth_author_add_diagram`, `mauth_question_upsert`, or `mauth_author_ensure_solutions` call should receive that same direct tool for the retry instead of reopening the broad wrapper surface.
- `mauth.author.ensureSolutions` creates or resizes the matched student answer space and adds solution-only text blocks. It is intended for compact solution tasks where the current document summary already contains enough question text; do not inspect only to confirm ids or module counts.
- `mauth.solutions.writeAll` is the whole-test marking-key tool. Use it for "write all solutions", "make the solutions key", "check every question has a solution", and similar broad requests. The payload must cover every marked question, part, and subpart; the tool preserves shared diagrams, creates/resizes student spaces, validates hidden `[[marks:n]]` totals, and returns a layout-check result before success.
- `mauth.layout.check` is the document-wide layout/print-readiness inspector. Use it for "check layout", "print risk", "weird blank pages", "missing answer spaces", "solution spacing", or broad student/solution-copy review. It reports issues such as missing solutions, missing student spaces, rendered overflows, oversized diagrams, and print-risk items; repair those with the narrow formatting, solution, response-space, or diagram tool rather than hand-patching raw JSON.
- Fresh document-wide layout-check prompts are routed directly to the native `mauth.layout.check` tool without an OpenAI call. This avoids spending tokens or timing out on a deterministic first step; if the checker returns repairable issues, the next provider call receives the tool output as context and may choose the focused repair tool.
- `mauth.author.adjustResponseSpaces` resizes or adds student-only answer/working spaces for existing questions, parts, or subparts. It is intended for focused layout-space edits and should preserve existing question wording, diagrams, solutions, ids, and ordering.
- `mauth.format.apply` handles focused formatting/layout edits without rewriting question content. It supports page breaks before questions, parts, and subparts; diagram alignment; answer-space adjustment; moving existing modules; fitting solutions to the paired student space; and conservative spacing tidy.
- Provider context is now focused by request: the backend selects relevant rule brains from `configs/ai-brains/` and filters large document summaries to referenced question numbers where possible. This is intended to improve accuracy first while reducing unnecessary token spend.
- The frontend trims old assistant chat history before a fresh provider request, excluding stale tool-round-limit messages and keeping only recent compact messages. Attachment/source-question turns are stricter: the provider receives the current request and attachments as a clean turn so old diagram-repair context cannot make it target the wrong question. This avoids paying repeatedly for old failed attempts.
- The backend normalises common provider tool-call shape mistakes before the frontend sees them: invalid JSON becomes a structured parse error, nested JSON-string arguments are parsed, action arrays are wrapped as `{ "actions": [...] }`, and unwrapped action/file arguments beside `name` are preserved. This is a safety net only; prompts and tests should still prefer the canonical `mauth_tool` shape `{ "name": "mauth.actions.preview", "arguments": { "actions": [...] } }`.
- Provider HTTP errors should be converted to plain readable messages at the API boundary. Do not surface raw OpenAI JSON, missing-tool-output payloads, or stack-style provider details in the teacher-facing chat panel.
- Before `mauth.actions.preview` or `mauth.actions.apply` reaches the document action engine, the assistant tool boundary validates actual action payload fields. It checks required ids, scopes, placements, patch objects, marks, content block shapes, diagram graph configs, and document-action fields. Diagram validation is type-specific for the supported renderers: `graph2d`, `vector2d`, `graph3d`, `image`, `statsChart`, `geometricConstruction`, `network`, and `setDiagram`. Validation failures return repairable `validationIssues` paths such as `actions[0].blocks[0].lines`, `actions[0].blocks[0].graphConfig.data.chartType`, `actions[0].blocks[0].graphConfig.data.functions`, `actions[0].blocks[0].graphConfig.metadata.view3d.camera`, or `actions[0].patch.graphConfig.metadata.vector2d.vectors[0].components[1]`; no document mutation occurs.
- For generic circle-geometry proof authoring involving a tangent and an angle subtended at the circumference, the assistant should prefer the robust radius/central-angle proof path and avoid adding parallel-chord scaffolding unless the teacher asked for parallel lines. This is a question-quality rule, not a separate canned diagram recipe.
- Assistant-authored answer-space blocks must be student-only, and solution text blocks must be solution-only. The action validator rejects ordinary visible `Solution...` text blocks, answer-space blocks without `visibility: "student"`, and raw `[[marks:...]]` placeholders before preview/apply mutates the document.
- High-level question authoring supports artifact answer surfaces. For sketch/label/shade/draw-on-diagram and completion-table tasks, use `answerSurface: "diagram"` or `answerSurface: "table"` with a blank/partial student diagram/table and a completed `solutionDiagram` or `solutionTable`. This keeps the student and solution copy layout fixed without adding a separate response-space block.
- Completed solution tables and diagrams now receive surface mark ticks from the relevant marks, so red checks render beside the completed table/diagram rather than on a separate explanatory text line.
- For artifact answer surfaces, any `solutionText` on the same question or part as a completed `solutionDiagram` or `solutionTable` must be an unmarked note. Do not duplicate those item marks with hidden `[[marks:n]]` ticks in the text.
- High-level question authoring also rejects structured parts with blank prompts or prompts that contain only a visible label such as `(a)`, so screenshot conversions cannot silently create marked empty part rows.
- If an attachment/source-question conversion accidentally calls `mauth.author.addDiagram` for a non-existent next question, the validation error points back to `mauth.question.upsert`/`mauth_convert_source_question`, and the backend repair round exposes the source-question tool instead of repeating the diagram-only tool. This is the intended repair path for prompts like "add this question to the test".
- If an attachment/source-question conversion reaches `mauth.question.upsert` but fails on a diagram schema path such as `graphConfig.metadata.vector2d.vectors[0].id` or `.start`, the repair remains on `mauth_convert_source_question` so the model fixes the whole new-question payload. It should not switch to `mauth.author.addDiagram`, because the question still has not been created.
- The action boundary also normalises assistant solution-mark mistakes across both high-level and low-level edits. If a solution-only text block contains visible `[1 mark]`, `(1 mark)`, `Solution (5 marks)`, or `\text{[1 mark]}` notes, the boundary rewrites them to hidden `[[marks:n]]` annotations before the renderer sees the document. This is a guardrail, not the desired authoring style; provider prompts and brains should still produce the hidden annotations directly.
- Before `mauth.files.*` tools reach the project-file driver, the assistant file-tool boundary validates file-operation payload fields. It checks required paths/names, file content for save/save-as, multi-path arrays, folder targets, rename/version fields, metadata objects, and revision fields. Validation failures return repairable `validationIssues` paths such as `arguments.path`, `arguments.paths[1]`, or `arguments.content`; no file operation runs.
- The Assistant chat composer accepts pasted, dropped, or file-picker attachments for images/screenshots, PDFs, Word `.docx` files, and text-like files such as `.txt`, `.md`, `.csv`, `.tsv`, `.json`, and `.tex`. The frontend limits the current request to four attachments, 10 MB per file, and 20 MB total, shows attachment chips in the chat, and warns that attachments can increase API cost. The backend passes images to the Responses API as `input_image` data URLs, PDFs as `input_file` data URLs, and Word/text-like files as extracted `input_text` on the latest user message.
- Attachment-derived questions should keep visible part tasks as structured part text. A source like `a) a · b`, `b) a · d`, `c) c · d` should become three parts whose prompts contain those scalar products, with the listed mark split applied to the matching parts.
- The backend reads provider token usage and returns a per-request estimated cost summary. The frontend attaches that summary as a small footer to the assistant reply for the whole teacher prompt, including internal tool-loop calls. Keep pricing tables backend-owned, treat them as estimates, and update them from the official provider pricing page when models change.
- `pnpm eval:assistant:local` runs zero-cost local fixture assertions against the same semantic graders used by the paid live eval script. It covers representative real-exam payloads and known-bad diagram shapes for Methods coordinate graphs, Specialist statistics charts, slope fields, Argand/locus regions, graph3d prisms, and implicit curves. Use it for tight looping before spending provider credits.
- `pnpm eval:assistant:live` runs a small live provider eval for the focused teacher prompt "write Question 1 with a circle-geometry proof and solution". `pnpm eval:assistant:live:core` is the normal paid regression suite: it covers circle-geometry question writing, Penrose diagram follow-up, mark-allocation edits that must preserve diagrams, wording rewrites that must preserve diagrams, multipart probability authoring, and a Year 9 linear-intersection graph question that must append Question 2 and use two straight-line `graph2d` functions under a default $1.50 safety cap. `pnpm eval:assistant:live:diagrams` is the paid renderer-selection suite for `graph2d`, `setDiagram`, `statsChart`, and `vector2d` under a default $1.00 safety cap. `pnpm eval:assistant:live:repair` is the paid repair suite; it simulates validation failures for wrong diagram renderer choices and checks that the provider retries with the correct native graphConfig under a default $0.75 safety cap. `pnpm eval:assistant:live:attachments` is the paid attachment suite; it currently checks compact PDF-source, Word-source, and screenshot scalar-product conversion under a default $0.50 safety cap. `pnpm eval:assistant:live:real-exams:core` checks the first real-exam set for Methods coordinate graphs, Specialist geometricConstruction, and Specialist stats charts; `pnpm eval:assistant:live:real-exams:extended` checks harder Specialist slope-field, Argand, graph3d prism, and implicit-curve cases. `pnpm eval:assistant:live:confidence` is the bounded paid confidence pass for the highest-value teacher workflows: whole-test solution writing via `mauth.solutions.writeAll`, document-wide `mauth.layout.check`, and one repair round from layout warnings. `pnpm eval:assistant:live:all` runs every live case. These are paid real-provider checks, so use them deliberately.
- `pnpm smoke:assistant:self` runs a free deterministic assistant self-smoke suite. It puts the assistant in realistic teacher-request situations, executes the same high-level Mauth tools the provider uses, then grades the resulting document for behaviours such as appending the next missing question without refusal, preserving diagrams during mark edits, using native Penrose Substance for geometry follow-ups, rejecting wrong renderer choices for scalar-product and coordinate-vector prompts, sanitising visible mark notes, targeting multipart solutions, writing whole-test solutions while preserving diagrams, layout-checking missing answer surfaces/oversized diagrams, rejecting malformed diagrams, and running validation after authoring.
- Repair evals should model the exact frontend loop: first tool call, failed tool output with `validationIssues`, then one provider retry. Add a repair eval when a failed validation class should become an automatic correction rather than a teacher-visible error.
- Provider keys are backend-only. Put `OPENAI_API_KEY` in the repo-root `.env` or `apps/api/.env`; do not expose it through Vite/frontend environment variables. `OPENAI_MODEL` is optional.
- `pnpm test:web-actions` covers the action engine and assistant tool dispatcher.

## Required Chatbox Loop

1. Load the relevant brains for the request: question, formatting, diagram, and/or solutions.
2. Use focused high-level tools when they fit the request: `mauth.question.upsert` for one-question writing, `mauth.author.addDiagram` for diagram follow-ups, `mauth.author.ensureSolutions` for compact solution-writing tasks, `mauth.solutions.writeAll` for whole-test marking keys, `mauth.layout.check` for document-wide layout/print readiness checks, `mauth.author.adjustResponseSpaces` for focused answer-space edits, and `mauth.format.apply` for focused formatting/layout edits.
3. Call `mauth.preview.inspect` for current/selected question work, especially when checking diagrams, answer-space layout, solution replacement slots, hidden tick totals, or blank part warnings. For Penrose circle-theorem diagrams, treat `semanticWarnings` as repair targets before saying the diagram matches the question. If the browser preview is mounted, the tool also reports rendered page occupancy, selected anchor bounding boxes, diagram render failures, solution-slot fit, and L-shaped diagram/answer-space metrics. Call `mauth.document.inspect` before broader edits or whenever the current compact summary lacks enough whole-document context.
4. For broader edits, generate an atomic batch of structured Mauth actions.
5. Call `mauth.actions.preview`.
6. If preview or commit preflight fails, repair the action batch using the returned `validationIssues` rather than patching raw state.
7. For solution/test-generation work, call `mauth.validation.run`; for broad solution-key or print-readiness work, also use `mauth.layout.check` in the relevant mode.
8. Call `mauth.actions.apply` only after the preview and validation are acceptable.
9. Commit the returned document through editor history/autosave.
10. Report what changed using ids, counts, and validation status.

The in-app chatbox UI/provider adapter must call `runMauthAssistantAdapterTool`. The lower-level document/file tools remain useful for tests and specialised workflows, but the adapter is the normal boundary because it owns the handoff from tool results to host editor/file callbacks. Provider tool calls should translate to this adapter path, not introduce a second editing path.

## Parity Roadmap

1. Harden the current chat loop: progress labels, robust continuation, clean provider errors, schema validation, retry/repair loops, and usage/cost summaries.
2. Make preview/document inspection closer to what Codex sees: selected item, nearby modules, document structure, rendered page metrics, validation state, and file state in compact tool results.
3. Add renderer tools: page/layout inspection, solution-slot fit inspection, diagram screenshot/metadata inspection, and print/PDF-specific checks.
4. Add attachment tools: pasted images/screenshots, PDFs, Word documents, and text-like source files are now routed through backend-owned provider calls; next add persistent storage, extraction caching, source metadata, and explicit upload/retention controls.
5. Add high-level authoring tools for common teacher prompts. Current examples are `mauth.question.upsert`, `mauth.author.addDiagram`, `mauth.author.ensureSolutions`, `mauth.solutions.writeAll`, `mauth.layout.check`, `mauth.author.adjustResponseSpaces`, and `mauth.format.apply`; next examples include repair-all solution spaces, whole-source conversion, more diagram families, print check, combine tests, and generate marking key variants.
6. Add project-level intelligence: search files, compare tests, duplicate/rename/move safely, and later integrate question-bank and curriculum-bank retrieval.
7. Add an evaluation set: saved tests and prompts that compare in-app assistant outcomes against expected document changes, validation status, visual screenshots, and print behaviour.
8. Keep the deterministic `pnpm smoke:assistant:self` suite and zero-cost `pnpm eval:assistant:local` fixtures as the first cheap gates before paid provider evals. Add scenarios whenever the assistant makes a new class of mistake, then fix the tool contract or brain rules until the scenario passes.
9. Run focused cost/performance passes only after the relevant evals pass. The optimisation order is: remove unnecessary provider calls with native tools, shrink document/context payloads, retrieve only the needed brains, narrow exposed tools, cache stable context, and then consider cheaper routing models where they do not reduce quality.

## Guardrails

- The chatbox should prefer actions over raw JSON mutation whenever an action exists.
- Raw document patches are acceptable only as a temporary development fallback and should become a new Mauth action if repeated.
- Mauth actions and solution validation should stay background systems. Teachers should interact with plain chat, normal editor controls, and understandable activity summaries rather than raw action JSON or validation-control buttons.
- AI-created tests should include generous student space, concise solutions, mark ticks, and validation.
- Student-space blocks should remain the replaceable surface for solution copies; ordinary question modules should be shared between student and solution output.
- File operations must use the project/file API, not browser cache or local component state.
- File saves must use the loaded `baseRevision`; on conflict, ask the teacher to reload or save as a copy rather than overwriting disk changes.
- Keep file/revision/autosave status quiet. Show short, human messages for completed work; do not expose raw file-tool JSON, revision plumbing, or repeated autosave notices to the teacher.
- External LLM use needs explicit provider settings, backend-only API keys, visible model/action logs, and clear user control over what content is sent.
